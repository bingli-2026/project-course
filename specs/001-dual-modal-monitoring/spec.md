# Feature Specification: 软硬件WebAI一体化设备状态监测

**Feature Branch**: `001-dual-modal-monitoring`  
**Created**: 2026-05-07  
**Status**: Draft  
**Input**: User description: "我们要实现这个doc中描述的项目"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 完成双测点数据采集与状态判别 (Priority: P1)

作为课程项目小组成员，我需要在一次实验中同时获取传感器与视觉两类数据，并得到设备状态判别结果，以证明系统具备最小可用监测能力。

**Why this priority**: 这是项目最核心价值，直接对应课题目标和中期/结题演示的最小闭环。

**Independent Test**: 在固定实验对象上执行一次标准采集流程，系统可输出带时间窗口标识的双模态样本与对应状态判别结果，且全流程可重复执行。

**Acceptance Scenarios**:

1. **Given** 已完成实验对象与采集链路准备，**When** 用户发起一次标准采集任务，**Then** 系统生成包含传感器特征、视觉特征和统一窗口标识的样本记录。
2. **Given** 已生成双模态样本记录，**When** 用户执行状态识别，**Then** 系统返回明确的状态类别及可核验的结果摘要。

---

### User Story 2 - 通过Web界面进行过程管理与结果查看 (Priority: P2)

作为指导教师或答辩评审，我需要通过可视化页面查看采集过程、状态结果和关键特征摘要，以便快速理解系统有效性。

**Why this priority**: 项目最终交付需要可演示性，Web 展示是将算法与系统成果转化为可沟通结果的关键。

**Independent Test**: 用户无需命令行即可在页面上完成一次任务发起、进度查看和结果浏览，且页面数据与底层输出一致。

**Acceptance Scenarios**:

1. **Given** 系统处于可用状态，**When** 用户在页面发起采集与识别任务，**Then** 页面显示任务状态变化直至完成。
2. **Given** 任务已完成，**When** 用户查看结果页面，**Then** 页面展示状态结论、关键特征摘要与对应时间窗口信息。

---

### User Story 3 - 支持少样本新增工况的增量更新 (Priority: P3)

作为项目小组成员，我需要在新增少量样本后完成模型更新并保留历史能力，以证明系统具备工况扩展能力。

**Why this priority**: 这是课题创新点之一，但依赖 P1 的基础数据与识别闭环，因此优先级低于 P1/P2。

**Independent Test**: 在引入一类新增工况的小样本后，系统可完成一次更新流程，并输出更新前后对比报告。

**Acceptance Scenarios**:

1. **Given** 已存在可用基线模型，**When** 用户提交新增工况小样本并触发更新，**Then** 系统完成更新并记录版本变化。
2. **Given** 更新已完成，**When** 用户运行对比评估，**Then** 系统输出新增工况识别表现及对历史工况影响的对比结果。

### Edge Cases

- 当某一模态在本次采集中缺失或质量不足时，系统如何标记该任务并避免输出误导性结论？
- 当同一时间窗口内两模态时间戳无法对齐时，系统如何处理该窗口并保留审计记录？
- 当新增样本量低于最小更新阈值时，系统如何拒绝更新并给出可操作反馈？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持对一次实验任务同时采集并管理传感器数据与视觉数据。
- **FR-002**: 系统 MUST 将两类数据按统一时间窗口组织为可追踪样本，并为每个窗口分配唯一标识。
- **FR-003**: 系统 MUST 基于双模态样本输出设备状态类别结果，并提供可读结果摘要。
- **FR-004**: 系统 MUST 提供 Web 端任务发起、任务状态查看与结果浏览能力。
- **FR-005**: 系统 MUST 为每次任务记录关键元数据（任务时间、样本规模、状态结论、结果版本）。
- **FR-006**: 系统 MUST 支持对新增少样本工况执行增量更新流程，并保留更新前版本以便回溯。
- **FR-007**: 系统 MUST 在更新后输出新增工况与历史工况的对比评估结果。
- **FR-008**: 系统 MUST 在任一关键步骤失败时给出可操作的错误反馈，并记录失败原因。

### Key Entities *(include if feature involves data)*

- **ExperimentTask**: 一次完整实验任务，包含任务标识、创建时间、当前状态、执行日志和结果引用。
- **TimeWindowSample**: 统一时间窗口样本，包含窗口标识、双模态特征集合、数据质量标记与标签信息。
- **StateInferenceResult**: 一次识别输出，包含状态类别、置信摘要、结果时间和关联窗口范围。
- **ModelVersionRecord**: 模型版本记录，包含版本号、训练样本范围、更新时间、更新类型与回滚引用。
- **IncrementalUpdateReport**: 增量更新报告，包含新增样本概况、更新前后对比指标、风险提示与结论。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% 的标准演示任务可在单次流程内完成“采集-识别-展示”闭环，无需人工补录数据。
- **SC-002**: 至少 90% 的有效时间窗口可成功完成双模态对齐并进入识别流程。
- **SC-003**: 项目组成员可在 5 分钟内通过 Web 页面完成一次任务发起并查看最终结果。
- **SC-004**: 在新增工况小样本更新后，新增工况识别成功率相对更新前提升至少 20%，且历史工况整体表现下降不超过 5%。

## Assumptions

- 实验环境可稳定提供目标设备运行与重复采样条件。
- 项目范围内的状态类别数量保持在课程计划约定范围内，不在本特性中扩展到大规模工业场景。
- Web 使用者为课程团队与指导教师，不涉及复杂多租户权限场景。
- 每轮增量更新均具备最小可用样本量与基础标注质量。
- 本特性优先保障课程演示与研究验证价值，不包含生产级高可用与大规模并发目标。
