## 每人契约

### 冶秉礼 — 导师对接、设备驱动、相机数据采集
- **产出**:原始相机视频文件;STM32 ↔ Orange Pi 通信驱动
- **消费**:来自甘芸瑄 / 杨喆 的硬件
- **交付给**:张展硕(视觉处理需要原始视频)
- **仓库位置**:`src/project_course/camera/`(已有相机 CLI 工具)
- **关于"back-end driver"**:这里指**设备侧驱动软件**(STM32 到 Orange Pi 之间),**不是** Web 后端。为避免混淆,在 commit 信息和聊天里建议用 *"acquisition driver / 采集驱动"* 这个说法。

### 张展硕 — 视觉数据处理
- **产出**:每个样本一个 CSV,包含 `vision_*` 列,符合 [`data-contract.zh.md`](data-contract.zh.md) 和 [`feature_schema.md`](feature_schema.md)
- **消费**:冶秉礼采集的原始视频
- **交付给**:贺鑫皓(把文件丢进 `data/samples/`)+ 杨炫志(同一份文件)
- **仓库位置**:视觉处理脚本当前在 `laboratory/` 下;长期应迁移到 `src/project_course/vision/`

### 传感器处理 — 暂无人接手(见下方 Open Q §1)
- **产出**:每个样本一个 CSV,包含 `sensor_*` 列
- **消费**:STM32 → Orange Pi 的传感器数据流
- **现状**:`feature_schema.md` 写明"sensor branch extraction: pending"。在有人接手之前,仪表盘上每个真实样本的 `has_sensor` 都会是 `false`。

### 杨炫志 — 模型训练
- **产出**:训练好的模型文件(根据 2026-05-10 与贺鑫皓的澄清,**不**由 Web 后端提供推理服务;模型跑在边缘端)
- **消费**:`data/samples/*.csv`(和仪表盘读的是同一批文件)
- **交付给**:Orange Pi 上的边缘运行时
- **可选(后期)**:把 `predicted_label` / `prediction_confidence` 列回写到 CSV(数据契约里已预留)
- **仓库位置**:暂定 `src/project_course/modeling/`

### 贺鑫皓 — Web 前后端(本仓库)
- **产出**:`http://localhost:5173` 的 Web 仪表盘 + `http://localhost:8000/api/v1` 的 REST 接口
- **消费**:`data/samples/*.csv`(只读)—— **不**回写
- **仓库位置**:`src/project_course/api/`、`frontend/dashboard/`
- **不负责**:模型推理、实时流、增量学习、原始视频的存储

### 甘芸瑄 / 杨喆 — 硬件
- **产出**:能跑的物理装置(电机 + 减速箱 + 加速度计 + STM32 板);BOM / 接线文档
- **消费**:经费
- **交付给**:冶秉礼(把硬件交给他去采集数据)
- **仓库位置**:硬件笔记、原理图、BOM 待新建 `doc/hardware/`

### 张嘉欣 — 文档与测试
- **产出**:开题 / 中期 / 结题报告;测试计划;答辩 PPT
- **消费**:其他所有人的产出
- **仓库位置**:`doc/`

## 文件系统边界

| 路径 | 拥有者 | 谁会读 |
|---|---|---|
| `src/project_course/api/` | 贺鑫皓 | — |
| `frontend/dashboard/` | 贺鑫皓 | — |
| `data/samples/` | 特征生产方(张展硕 + 传感器负责人) | 贺鑫皓(Web BE)、杨炫志(模型) |
| `data/project_course.sqlite3` | Web BE 自动生成 | —(不要手动改) |
| `src/project_course/camera/` | 冶秉礼 | — |
| `laboratory/` | 张展硕(视觉原型) | 仅供参考 |
| `doc/` | 全员(走 PR review) | 全员 |

**不要直接写 `data/project_course.sqlite3`。**它是后端每次启动时根据 `data/samples/` 重建的索引文件。要清空就直接删掉它,不会有事。

## Open Questions(阻塞协作)

下面 5 件事,如果不答清楚就会导致返工。已经放进群消息里准备发了:

1. **谁负责传感器侧的特征提取?** `feature_schema.md` 声明了 30+ 个 `sensor_*` 列,但没人的角色明确包含这个。建议在 2026-05-19 之前产出第一份传感器 CSV,这样还有时间做融合再到中期。

2. **冶秉礼的 "back-end driver" 是设备驱动还是 Web 后端?** 如果是 Web 后端 —— 和贺鑫皓的工作有重叠,需要重新切分。(当前假设:仅设备驱动。)

3. **杨炫志的模型在中期演示时跑在哪?** 贺鑫皓的工作假设(基于 2026-05-10 的对话):推理跑在边缘端,Web 仪表盘只做展示。如果模型其实要由 Web 后端来跑,数据契约里 `predicted_label` 就要变成必填,后端还得加新路由。

4. **Label 词表**是 `normal / unbalance / loose / misaligned / unknown`(见 `data-contract.zh.md`)吗?数据集和模型准确率都依赖于这个集合是闭合且稳定的。

5. **窗口长度**两边都固定 0.5s 吗?`doc/context.md` 提了滑窗但没定数值。生产方必须在生成"真实"数据之前商量好,否则 vision 和 sensor 行没法融合。

可以直接复制到群里的对齐消息见 [`team-alignment-questions.zh.md`](team-alignment-questions.zh.md)。
