> 中期演示是 5 月 26 号,在我们各支线全力冲刺之前,我觉得有 5 个决定得先锁一下,不然视觉 / 传感器 / 模型 / Web 这几条线很可能默默走偏,到时候集成对不上就要返工。我这边写 Web 仪表盘是先按一组假设来的,详见仓库里的 `doc/data-contract.zh.md` 和 `doc/team-interfaces.zh.md`。如果谁的假设跟我不一样,**现在**就说,比集成的时候说成本低很多。

> **1. 传感器侧特征提取由谁来做?**
> Schema 里声明了 ~30 个 `sensor_*` 列(见 `doc/feature_schema.md`),但目前没人的分工明确包含"把 STM32 → Orange Pi 的传感器流转成这些列"这件事。没有这一步,我们所有"真实"样本都会是只有视觉、没有传感器,中期那个双模态故事就讲不出来了。建议:5 月 19 号之前产出第一份传感器 CSV。
>
> **2. 冶秉礼说的 "back-end driver" 是设备驱动还是 Web 后端?**
> Web 后端(FastAPI,在 `src/project_course/api/` 下)是我(贺鑫皓)在做。我的假设是 Bingli 说的 "back-end driver" 指**设备侧驱动**(STM32 到 Orange Pi 那一段),不是 Web 后端。Bingli 你确认一下 —— 如果其实是 Web 那侧,我俩有重叠要切一下。
>
> **3. 中期演示的时候,模型跑在哪里 —— 边缘还是 Web 后端?**
> 我现在的假设:杨炫志的模型跑在 Orange Pi(边缘端),Web 仪表盘**只负责展示和存储**,不做在线推理。如果模型其实要由 Web 后端按需调,那需要:
> - 加一个新接口(`POST /api/v1/predict`)
> - 数据契约里 `predicted_label` / `prediction_confidence` 要改成必填
> - 大概多花 1 周开发时间
>
> 麻烦确认一下,免得我做错方向。
>
> **4. Label 词表能定一个闭集吗?**
> 建议:`normal / unbalance / loose / misaligned / unknown`(全小写 ASCII,严格拼写)。筛选、训练标签、报告都依赖大家用同一套词。如果有人有更合理的分类法(论文里的),现在提。
>
> **5. 窗口长度固定 0.5 秒吗?**
> 视觉行和传感器行只有在两边用同一套滑窗定义的时候才能融合。开题报告说要用 Welch 滑窗但没定时长。建议两边都固定 0.5 秒、0.25 秒 hop。视觉组 / 传感器组,你们的采样率(视觉 420 fps、传感器 ~1.6 kHz)用这套窗合理吗?
>

## 为什么是这 5 个、不是更多

其实还有很多**可以**讨论的东西(增量更新接口、演示脚本、数据打标工具、等等),但那些这周不阻塞并行干活。上面这 5 个的特点是:**任何一个不答清楚,都会让某个人正在做的工作被默默作废**,所以放在群消息里;其他的就不放。

如果想后面再讨论别的,加到"下次同步会议"的议程,**不要**一股脑全堆到这次群消息里,否则这 5 个也没人答了。
